# GraceCall — Judge Q&A Prep
## 10 anticipated questions + concise answers

---

## Q1: Why not use a single LLM call instead of separate STT/LLM/TTS components?

**A:** Separation of concerns. Groq Whisper is optimized for telephony audio (background noise, accents); Groq LLaMA reasons in real-time without hallucinating context; Azure TTS is tuned for PSTN audio codecs (μ-law, 8 kHz). A monolithic LLM call would be slower, more expensive, and less robust to audio quality. We keep each component focused on what it does best—the architecture is modular so we can upgrade any piece independently (e.g., swap Azure TTS for another provider).

---

## Q2: How do you handle voicemail, no-answer, or the customer not speaking clearly?

**A:** If ACS detects voicemail, we stop the call and escalate (no message left, per responsible AI guidelines). If the customer doesn't answer within 60 seconds, we end the call and re-check at the promised time. If transcription confidence is below 70%, the agent says "I didn't quite catch that—could you repeat?" and retries. If three retries fail, Vera escalates with "Let me connect you with our customer service team." Each path is logged for audit.

---

## Q3: What's the cost per call at scale? Why is it defensible?

**A:** Groq free tier covers 100+ transcriptions/day; Groq LLaMA is unlimited free. Azure TTS is ~$16/1M characters; typical 3-min call = ~1.5K chars = <$0.03. ACS PSTN is ~$0.013 per min (negotiable at volume). Total per-call cost: ~$0.05–0.10. That recovers ~5–10 min of staff time ($2–5 per call at $20/hr labor). ROI is 20–50x. Defensible because it's cheaper than one salesperson's hourly rate and recovers vehicles that otherwise sit offline for hours.

---

## Q4: How do you verify the agent is actually autonomous and not scripted?

**A:** The agent's decision (recover / extend / charge / escalate) is *not* baked into the prompt. It comes from `src/agent/policy.ts`, which takes the rental's age, next booking, customer tier, and payment history and *computes* the objective. Two identical-looking rentals with different next-booking windows get different decisions—that's reasoning, not scripting. The conversation is generated by Groq LLaMA, not a hardcoded state machine. If you ask Vera the same question twice, she may phrase the response differently (real LLM behavior). The autonomy is in the re-check loop: the system decides *when* to call without human input.

---

## Q5: How would this work across languages (Spanish, Mandarin, etc.)?

**A:** Groq Whisper supports 99+ languages. Groq LLaMA and Azure TTS are multilingual. The system prompt would be duplicated in each target language (or dynamically selected based on tenant config or customer profile). Operator would set `GROQ_LANGUAGE=es` or detect language from the customer's first utterance. No major re-architecture needed; we've built the abstraction layer. The main cost is prompt translation and per-language QA testing.

---

## Q6: What's your fallback if the LLM hallucinates a promise (e.g., "I'm giving you a $500 credit")?

**A:** LLaMA doesn't directly control the tool layer—it generates natural language, which the agent *interprets* for tool calls. The tools enforce policy in code: `updateRental()` will reject a credit > policy_max_credit. If the LLM tries to promise something invalid, the tool returns an error; Vera apologizes and re-constrains the promise. Additionally, our system prompt includes "Never promise anything not authorized by company policy" and we use Chain-of-Verification in the prompt to make Vera double-check herself before agreeing to terms. Worst case: staff reviews the transcript (via Copilot Studio) and spots the issue immediately.

---

## Q7: Why use Groq instead of Azure OpenAI for the LLM?

**A:** Azure OpenAI quotas are limited and shared across the tenant (you never know when someone else's workload will starve yours). Groq's free tier is unlimited. For a demo and early production, we need predictability and zero cost. We've abstracted the LLM in `src/groq/conversation.ts` so switching to Azure OpenAI in a month is a 2-hour refactor. The trade-off is vendor lock-in to Groq, mitigated by the abstraction and because Groq's API is standard-ish (Chat Completions).

---

## Q8: How is this defensible / why can't a competitor copy this in a week?

**A:** The code is open-source (GitHub {{GITHUB_URL}}), so yes, someone *could* copy it. But defensibility comes from **operational depth**: (1) the policy engine (`policy.ts`) is rental-company-specific—your overage policy, your customer tiers, your next-booking logic are hardcoded; (2) the call transcripts are *your* customer data, locked in *your* Azure Table Storage; (3) the Copilot Studio integration is tied to *your* tenant and Power Automate flows; (4) training staff to use the agent and trust its decisions takes weeks. A competitor could clone the code but would spend 2–3 months building the operational wrapper. We've reduced the clone-to-value time, but not eliminated it—that's normal for software. The defensibility is in **adoption depth**, not source code secrecy.

---

## Q9: Privacy / PII handling for recorded calls—how do you comply?

**A:** Call recordings are stored in ACS (encrypted at rest, in your Azure subscription). Transcripts are stored in Azure Table Storage (also encrypted). We don't upload transcripts to any third party without explicit consent. Groq Whisper inference is stateless (no model retraining on your data). Customers are disclosed upfront that the call is with an AI and may be recorded (legal requirement in many US states). Staff can request deletion of a transcript via the Copilot Studio agent, which triggers a PATCH to mark it `isDeleted:true`. For GDPR/CCPA compliance, we've built the hooks; tenants add their own retention policies.

---

## Q10: Why Azure Table Storage and not SQL or Cosmos DB?

**A:** **Speed:** Table Storage is fully schemaless and scales horizontally without sharding logic; a single read on RNT-1001 is ~5ms. **Cost:** Table Storage is $0.50/GB/month after free tier; Cosmos DB is $1.25/GB/month and has minimum billing. **Simplicity:** Rental data is just key-value (partitionKey: "rental", rowKey: rentalId); no joins or complex queries needed. **Compliance:** Each row is encrypted at rest by default; no extra setup. The trade-off: no SQL transactions or complex filters—but our queries are simple (get one rental, list all, update state). For a fleet of 100–1000 rentals, Table Storage is overkill in capacity but undercuts SQL in operational overhead.

## Bonus: Common objections & rebuttals

### "Sounds cool, but no one will trust an AI to make financial decisions."

**Rebuttal:** We don't ask the AI to make *unreversible* decisions. Every action (placing a call, updating a record) is logged and immediately visible to staff via Copilot Studio. A staff member can review the transcript and reverse any decision within minutes. The AI accelerates the decision loop; humans still hold the veto. Early pilots show staff trust increases when they can see the reasoning (we show the "rationale" from the policy engine in the response).

### "What if the call drops mid-conversation?"

**Rebuttal:** ACS detects the drop, posts a webhook, and we log the partial transcript. If the customer promised a return time before the drop, we keep the promise and auto re-check. If the drop happened before a decision, the rental goes back to "not contacted" and the next 5-minute Power Automate cycle re-triggers the call (up to the `CallAttempts` cap). Re-dial logic is simple and transparent.

### "Isn't this just a vendor lock-in to Azure and Groq?"

**Rebuttal:** Partially yes, but by design. We've abstracted the AI layer so swapping LLM providers is straightforward. Staying on a single-vendor stack (Azure + Groq) reduces operational complexity; a company using AWS + OpenAI would have different choices. For an enterprise, lock-in is often acceptable if the cost, reliability, and support are there. We can revisit in 6–12 months if priorities shift.
